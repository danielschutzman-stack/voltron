1 · Session Bootstrap
🚀
How Every Session Starts
Runs automatically on first message — takes ~15–20 seconds
💬 Plain English

Think of it like a contractor picking up their tools before starting the job. Voltron is smart, but it doesn't permanently carry around all the specific instructions it needs to talk to ThoughtSpot, look up accounts, or structure research tasks. Those live in separate files stored on GitHub — a free, public file-hosting service.

When you start a session, Voltron spends about 15–20 seconds fetching those files and loading them into its working memory. After that, it's fully equipped for everything — pulling territory data, running PG plans, launching research agents — for the rest of the session.

Why store the files externally instead of building them in permanently? Because it means we can fix bugs or add new features just by editing a file on GitHub. No reconfiguring Voltron, no IT tickets, no downtime. When the territory pull was broken on March 16th, we updated two files on GitHub and every AE got the fix automatically on their next session.

⚙️ Technical Detail

Every Voltron session begins with a 4-step bootstrap sequence that wires up the Python query infrastructure. It runs once at session start and is skipped if files already exist in the sandbox.

  1. Fetch 4 files concurrently from GitHub Gists — bootstrap.py, ts_intent_map.py, ts_runner.py, and subagent_templates.py fetched in parallel (~4–6s).
  2. Write and execute bootstrap.py — confirms the loader script is functional before proceeding.
  3. Call bootstrap() with the 3 helper file contents — ts_intent_map.py + ts_runner.py are concatenated into /sandbox/ts_fallback_map.py; subagent_templates.py written to /sandbox/subagent_templates.py. Files skipped if they already exist.
  4. Session declared ready — agent confirms setup and presents the opening question.

2 · Query Infrastructure
🔌
How ThoughtSpot Data Is Fetched
run_with_fallback() · ThoughtSpot REST API v2 · 4 worksheets
💬 Plain English

When Voltron needs data from ThoughtSpot — your accounts, deal stages, MEDDPICC scores, activity history — it sends a structured question directly to ThoughtSpot's database and gets back rows of data, just like a search.

Every query goes through a smart wrapper called run_with_fallback(). Here's what makes it smart:

  • It has a backup plan. If the first query comes back empty, it automatically tries a slightly different version of the same question against a different data source — without you ever knowing it retried.
  • It knows when your session has expired. ThoughtSpot tokens expire after ~60–90 minutes. If a query hits an expired token, it immediately surfaces a warning rather than silently failing.
  • It times out gracefully. If ThoughtSpot takes more than 20 seconds to respond, it stops waiting and reports the issue — it never hangs indefinitely.

There are four ThoughtSpot data sources (called worksheets) Voltron can query. Most territory and deal data lives in the GTM RevOps worksheet. Stakeholder and renewal data lives in studio GTM RevOps. Voltron picks the right one automatically based on what you're asking for.

⚙️ Technical Detail

The Two Core Functions

_ts_query(query_string, worksheet_key, timeout=20) — Low-level. Fires POST to /api/rest/2.0/searchdata, handles auth, timeout, HTTP errors. Never called directly by the agent.

run_with_fallback(intent, **variables) — High-level. Looks up intent in INTENT_MAP, formats query template, fires primary query, auto-retries fallback on empty/error. Always use this.

3 · Intent Map
🗺️
The 12 Query Types
Each intent = a named question Voltron knows how to ask ThoughtSpot
💬 Plain English

Rather than writing a new database question from scratch every time, Voltron has a library of 12 pre-built query types — called "intents." Each one is a named, reusable question with a blank to fill in, like: "What are all the accounts owned by ___?" or "What does the MEDDPICC scorecard look like for ___ account?"

Each intent also has a backup version. If the first attempt comes back empty, Voltron automatically tries the backup — which might ask the same question slightly differently, or pull from a different data source. This happens invisibly and instantly.

There are two types of intents: ones that pull data for a whole territory (filtered by the AE's name) and ones that pull data for a specific account (filtered by account name).

The newest intent — territory_overview — is the star of the show. It replaced five separate questions with one, cutting territory pull time from ~40 seconds down to ~2 seconds.

⚙️ Technical Detail

Each INTENT_MAP entry is a 4-tuple: (primary_ws, primary_template, fallback_ws, fallback_template). Templates use Python .format() with named variables.

4 · Territory Workflow
🗂️
Territory Workflow — End to End
Trigger: "Explore my territory" or "pg" → territory path
💬 Plain English

The territory workflow is designed to get an AE from "I want to see my accounts" to "I have a prioritized list and research on my top 5" as fast as possible.

Here's how it flows in plain terms:

  • Step 1 — Voltron looks you up automatically. Before asking you anything, it pulls your accounts from ThoughtSpot using your name. If it finds them, it skips the "who are you?" question entirely.
  • Step 2 — You pick how to rank your top 5. By ICP score, by how long since the last activity, by 6Sense intent signal, by vertical — your choice. All data is already loaded from Step 1, so this is instant.
  • Step 3 — Fast Sweep runs automatically across all 5 accounts. No module selection needed. Voltron kicks off web research and case study matching in the background while simultaneously pulling SFDC stakeholder data and 6Sense signals from ThoughtSpot. These run in parallel, so it takes roughly the same time as doing one account.
  • Step 4 — You pick one account for a full deep-dive. The data from the Fast Sweep is already loaded — no repeat work. Six more research modules are available, and you can run all of them or just the ones you want.
  • Throughout — a master Excel file is kept up to date. It updates as each piece of research comes in, so you can download a live version at any point. You don't have to wait for everything to finish.

5 · PG Plan Builder
📋
PG Plan Builder — End to End
Trigger: "pg" or "PG a specific account"
💬 Plain English

The PG Plan Builder is Voltron's flagship workflow — it researches a single account as deeply as possible and turns everything into three ready-to-use deliverables.

It runs up to 10 research modules simultaneously. Some of these pull live data directly from ThoughtSpot (deal history, MEDDPICC, stakeholder map, 6Sense signals). Others spin up background research agents that browse the web, scrape job postings, and look up executive profiles — all at the same time.

The three outputs it always produces:

  • Internal PG Report — everything Voltron found, organized into a scannable HTML document for the AE. Includes the full stakeholder map, MEDDPICC scorecard with named individuals, competitive landscape, recommended case studies, and outreach sequences.
  • Customer-Facing One-Pager — a clean, professional HTML document safe to share with the prospect. No internal names, no deal stage language, no Voltron branding. Just: their challenges → how ThoughtSpot fits → proof points → a demo link.
  • Updated Territory Excel — the master workbook is updated with links to both documents and the PG Plan Status is marked complete.

One important safeguard: the one-pager is built from scratch for the external audience. Anything internal — stakeholder names, MEDDPICC terminology, deal stages, the AE's name — is never included.

6 · Subagent Architecture
🤖
What Runs Where and Why
Main thread vs. background subagents — the decision framework
💬 Plain English

Voltron can spin up background "research agents" — mini versions of itself that go off and do a specific task while the main conversation keeps moving. But not everything gets handed off to a background agent. There's a deliberate split.

ThoughtSpot data queries always run in the main thread — never handed off. Here's why:

  • They're fast (2–15 seconds) — handing off to a background agent would actually be slower due to the overhead of spinning one up.
  • They need live access to your ThoughtSpot session token. If the token expires mid-query, the main thread can warn you instantly. A background agent would only discover the failure after a long wait.
  • They need context from the conversation — like the exact account name you're asking about. Background agents are "blind" to the conversation history and need everything spelled out for them.

Web research tasks always run as background agents. These are genuinely slow (a full web sweep can take 60–120 seconds), they don't need ThoughtSpot access, and they can run in parallel across multiple accounts at the same time — which is where the real time savings come from.

The rule of thumb: if it touches ThoughtSpot, it runs in the main thread. If it browses the web, it runs as a background agent.

7 · Error Handling & Fallback Logic
🛡️
Error Handling & Fallback Logic
What happens when things go wrong
💬 Plain English

Voltron is designed to keep going even when individual pieces fail. A missing data source or broken query never stops it from producing a report — it just notes the gap and keeps working with what it has.

There are four things that can happen when Voltron asks ThoughtSpot for data:

  • ✅ Data comes back — great, used immediately.
  • 🔁 Empty response — Voltron automatically tries a backup version of the same question. If that's also empty, it notes "Data unavailable" in the report and moves on. The report still gets built.
  • ⚠️ Token expired — Voltron stops all ThoughtSpot queries immediately and tells you to refresh the page. It will not retry silently or keep failing. You refresh, it continues.
  • ❌ Unexpected error — noted in the report, everything else continues. A broken module never holds up the other nine.

The same philosophy applies to web research: if LinkedIn blocks a job search, the agent immediately switches to a different source. If a company website is down, it notes it and moves on. Voltron never fabricates data to fill a gap — it just says what it couldn't find.

⚙️ Technical Detail

Status Codes from run_with_fallback()

StatusMeaningAgent Action
okData returned — column_names + data_rows populatedUse data for synthesis
empty0 rows from both primary and fallbackNote "[Module]: Data unavailable" — do NOT block synthesis
token_expired401/403 from ThoughtSpot APISurface ⚠️ refresh message, halt all TS queries immediately
errorHTTP error, timeout, or parse failureNote in report — do NOT block synthesis

Fallback Logic (step by step)

  1. Fire primary query (primary worksheet + primary template)
  2. If ok → return immediately
  3. If token_expired → halt, surface warning, stop
  4. If empty or error → fire fallback query (different worksheet/template)
  5. If fallback ok → return result with used_fallback: true
  6. If both fail → return empty or error — report gap, continue synthesis
8 · Token Management & Expiry
🔑
Token Management & Expiry
~60–90 minute session lifespan
💬 Plain English

To pull data from ThoughtSpot, Voltron uses a temporary access pass — called a token — that gets assigned at the start of every session. Think of it like a visitor badge at an office: it works all day, but it expires and can't be used tomorrow.

These tokens last roughly 60–90 minutes. For most sessions (a quick territory pull or a single PG plan), you'll never hit the limit. But for long working sessions — like doing a full territory sweep followed by multiple deep-dive PG plans — you might approach the expiry window.

Voltron handles this in two ways:

  • Proactive warning at 45 minutes: If you're doing something ThoughtSpot-heavy and the session has been running for 45+ minutes, Voltron will flag it: "Heads up — we're at 45 minutes and tokens expire around 60–90 minutes. If queries start failing, a page refresh will fix it."
  • Immediate halt if expired: If a query actually fails due to an expired token, Voltron stops all ThoughtSpot queries immediately and tells you exactly what to do. It won't keep trying and silently failing.

The fix is always the same: refresh the page. That generates a new token and everything picks back up.

9 · Deliverable Generation
📄
Deliverable Generation
Internal report · customer one-pager · territory Excel
💬 Plain English

Every PG plan produces three files automatically — no extra steps, no "would you like me to generate the report now?" prompts. As soon as research is done, Voltron synthesizes everything and generates all three at once.

  • Internal PG Report — a full HTML document for the AE's eyes only. Has everything: deal history, MEDDPICC scorecard, stakeholder map with Gong vs. SFDC cross-check, competitive landscape, relevant customer proof points, and personalized outreach drafts. It's designed to be used in deal prep, not shared with the customer.
  • Customer-Facing One-Pager — a clean, polished HTML document that can go straight to the prospect. Built from scratch with only public-safe content: their business challenges, how ThoughtSpot addresses them, relevant proof points, and a link to book a demo. Nothing internal ever leaks in.
  • Territory Excel — the master workbook is updated live throughout the session. Links to the PG report and one-pager are added, and the account's status is marked "PG Complete." You can download this at any point — it's saved after every update.

10 · How to Update Query Definitions
✏️
Updating Query Definitions via Gist
No prompt redeploy needed — changes take effect on next session for all users
💬 Plain English

If you need to fix a query, add a new data point, or change how Voltron pulls ThoughtSpot data — you edit a file on GitHub. That's it. No IT ticket, no configuration change, no redeploy of the agent.

The two files most likely to need changes are:

  • ts_intent_map.py — this is where the actual query definitions live. Change this if you're adding a new type of data pull, fixing a field name, or changing which worksheet a query goes to.
  • ts_runner.py — this is the underlying engine that fires queries and handles responses. Change this only if something structural about how ThoughtSpot responds has changed (rare).

The process: open the Gist, click Edit, make your change, click Update. On their next session, every AE automatically gets the updated version — no one has to do anything on their end.

One gotcha: GitHub's raw file URLs have a short-lived cache. After saving a Gist, the old version might appear for up to 5–10 minutes if you check the raw URL directly. The GitHub UI (the non-raw page) always shows the latest immediately. You can ask Voltron to verify by saying "check the gists."

Query Template Rules

  • All column names: [Column Name] bracketed format — no exceptions
  • Filter syntax: [Column Name] = 'value' — not unbracketed tokens
  • Max ~8 attribute fields per query on the 951-column RevOps worksheet
  • Avoid nested brackets: [Account Status [For Calculation]] causes parse errors
  • Always include a filter — open queries time out on large worksheets
